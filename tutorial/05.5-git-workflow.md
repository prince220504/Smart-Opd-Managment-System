## Half 1 — Why the Git location mattered

`git init` creates a `.git/` folder **in the folder where the command runs**, and everything inside that folder (recursively) becomes the repo.

An accidental `git init` from `C:\Users\Prince\` created `.git/` in the home directory. Symptoms:

```bash
git rev-parse --show-toplevel
# → C:/Users/Prince        ← wrong! this is the home folder
```

When we ran `git status` from the project, Git correctly walked up to find the nearest `.git/` and showed every untracked file in the home directory: Desktop, Downloads, AppData, NTUSER.DAT, personal files. One `git add .` would have staged everything.

### Fix

```bash
# 1. Delete the misplaced .git folder (it had zero commits, safe to remove)
rm -rf "C:/Users/Prince/.git"

# 2. Re-init inside the project root
cd "C:/Users/Prince/OneDrive/Desktop/Smart OPD Management System"
git init -b main

# 3. Verify
git rev-parse --show-toplevel
# → should print the project path, NOT C:/Users/Prince
```

The `-b main` flag sets the default branch to `main` instead of the legacy `master`.

## Half 2 — Commit identity

Every commit Git makes is signed with a name and email. To set them:

```bash
git config user.name "Prince"
git config user.email "your-github-email@example.com"
```

**Use the same email as your GitHub account** — that's how GitHub links commits to your profile and shows the green dots on your contribution graph.

- `git config <key> <value>` sets per-repo.
- `git config --global <key> <value>` sets across every repo on your machine.

Per-repo is fine if you only have one machine; global is convenient if you build many repos.

## Half 3 — `.gitignore` strategy

The `.gitignore` file tells Git which paths to **never track**, even if you accidentally `git add` them.

### What we ignore and why

| Pattern | What it covers | Why ignore |
|--------|----------------|-----------|
| `__pycache__/`, `*.pyc` | Python bytecode | Regenerated automatically; not source code |
| `.venv/`, `venv/` | Virtual environment | Rebuilt from `requirements.txt`; ~30 MB of files |
| `*.sqlite3`, `db.sqlite3` | Local SQLite DB | Different DBs for each dev; production uses PostgreSQL |
| `staticfiles/` | Output of `collectstatic` | Build artifact; regenerated on deploy |
| `media/` | User-uploaded files | Created at runtime; never in source |
| `.env`, `*.env` | Secrets (SECRET_KEY, DB password, email creds) | **Never commit secrets** — public repos are scraped within minutes |
| `Thumbs.db`, `.DS_Store` | OS metadata | Windows / macOS junk |
| `.vscode/`, `.idea/` | Editor settings | Personal preference, not project |
| `*.log`, `dist/`, `build/` | Build/log artifacts | Output, not source |

### What we DO track

Counter-intuitive but important: keep these IN Git:

- `CLAUDE.md` — the AI brief; future contributors (and AI tools) need it
- `opd_roles_and_final_structure.html` — the spec; reviewers need it
- `tutorial/` — the learning notes; portfolio value
- `requirements.txt` — pinned dependency list

### Golden rule for secrets

Anything in `.env` is a secret. Anything else is fine to commit. Pair `.env` (gitignored, has real values) with `.env.example` (tracked, has only key names, no values) so other devs know what to set.

## Half 4 — First commit + GitHub push

```bash
# Stage everything not ignored
git status            # see what will be staged
git add .
git status            # verify nothing unwanted is staged

# Commit
git commit -m "Initial commit: Django 6.0.6 project scaffold, settings wired to frontend/, tutorial steps 1-5"

# Create the repo on GitHub (web UI)
# - github.com → New repository
# - name: smart-opd-management-system (or your choice)
# - DO NOT check "Initialize with README/license/gitignore" — we already have those

# Connect and push
git remote add origin https://github.com/<your-username>/smart-opd-management-system.git
git push -u origin main
```

The `-u` flag in `git push -u origin main` sets `origin/main` as the **upstream tracking branch**. After this, future `git push` and `git pull` work without arguments.

## Half 5 — The empty folder gotcha

After the first push, we noticed `frontend/` was missing on GitHub even though it existed locally. The reason:

> **Git tracks files, not folders.** An empty folder has nothing to track, so Git silently skips it. There is no way to commit an empty directory.

### Two ways to handle this

**Option 1 — `.gitkeep` placeholders (works but adds clutter):**

Drop a zero-byte file named `.gitkeep` inside any empty folder you want preserved.

```bash
touch frontend/templates/accounts/.gitkeep
touch frontend/static/css/.gitkeep
# ...etc for every empty folder
```

Pro: GitHub immediately shows the full intended layout.
Con: Placeholder files have to be cleaned up later, and they have no real value.

The name `.gitkeep` is convention only — Git doesn't recognize it. Any non-ignored file works.

**Option 2 — Let folders appear naturally when content lands (industry standard):**

Don't create placeholder files. Empty folders simply don't exist on GitHub. When real templates from Stitch land in `frontend/templates/accounts/`, that folder appears on the next push.

Pro: No placeholder cruft. Cleaner repo. Folders earn their place by holding real content.
Con: GitHub doesn't show the full layout until each folder gets its first file.

### What we chose for this project — Option 2

We decided to keep things clean: no `.gitkeep` files. Folders will appear on GitHub when actual templates and CSS arrive in Steps 11–12 via Stitch + Antigravity.

**One small caveat:** `settings.py` references `frontend/templates/` and `frontend/static/` in `TEMPLATES['DIRS']` and `STATICFILES_DIRS`. If those paths don't exist *locally*, Django prints a warning (`staticfiles.W004`) when the server starts — not an error, just noise. To keep the dev experience clean we created the two top-level folders **locally only** (they remain empty, so Git still ignores them). When Stitch templates arrive, the subfolders (`accounts/`, `appointments/`, etc.) get created with real content and finally appear on GitHub.

If you ever do need to track an intentionally empty folder (e.g. a `logs/` directory the app expects to exist at runtime), use `.gitkeep` then — that's exactly the situation it was invented for.

## Half 6 — Industry branching workflow

From Step 6 onwards, **never commit directly to `main`**. The pattern:

```
main           ←  always working, deployed to Render via auto-deploy
   |
   +─── feature/accounts-app          (Step 6–9 work happens here, merges to main when done)
   +─── feature/appointments          (Steps 13 work here)
   +─── feature/lab-tests             (Step 15 work here)
   +─── feature/prescriptions         (Step 15 continues)
   +─── feature/notifications-celery  (Step 16 work)
   +─── chore/render-deploy           (Step 18 work)
```

### One feature branch per module

For each new app or feature:

```bash
# Start from updated main
git checkout main
git pull

# Create + switch to feature branch
git checkout -b feature/accounts-app

# Work, commit small logical steps
git add backend/apps/accounts/
git commit -m "feat(accounts): scaffold app via startapp"

git add backend/config/settings.py
git commit -m "feat(accounts): register apps.accounts in INSTALLED_APPS"

git add backend/apps/accounts/models.py
git commit -m "feat(accounts): add CustomUser with role field"

# Push branch to GitHub
git push -u origin feature/accounts-app
```

### When the module is complete and tested

Two ways to merge into `main`:

**Option A — Pull Request (recommended for portfolio look):**
1. On GitHub, click "Compare & pull request" for the feature branch.
2. Review your own diff.
3. Click "Merge pull request" → "Confirm merge."
4. Locally: `git checkout main && git pull` to sync.

This leaves a visible record in the GitHub UI. Looks great on portfolio.

**Option B — Local merge (faster, no PR trail):**
```bash
git checkout main
git pull
git merge feature/accounts-app
git push
```

After either, delete the merged feature branch:

```bash
git branch -d feature/accounts-app          # local delete
git push origin --delete feature/accounts-app   # remote delete
```

### Commit message convention — Conventional Commits

Format: `<type>(<scope>): <short description>`

| Type | Use for |
|------|--------|
| `feat:` | new feature |
| `fix:` | bug fix |
| `docs:` | docs/README/tutorial files |
| `refactor:` | restructure without behavior change |
| `chore:` | config, deps, tooling, folder structure |
| `test:` | adding/fixing tests |
| `style:` | formatting only, no logic change |

Examples:
- `feat(accounts): add CustomUser model with role field`
- `fix(appointments): correct slot overlap check`
- `docs(tutorial): write Step 7 - CustomUser`
- `chore: add .gitkeep placeholders for frontend folders`

### Branch naming

- `feature/<short-name>` — new modules (`feature/accounts-app`, `feature/appointments`)
- `fix/<short-name>` — bug fixes (`fix/login-redirect`)
- `chore/<short-name>` — non-feature work (`chore/update-deps`)

Use lowercase, hyphens for spaces, short and descriptive.

## Gotchas

- **Never commit secrets.** Public repos are scraped within minutes by bots looking for AWS keys, API tokens, DB passwords. If you accidentally commit `.env`, rotate the secret immediately and run `git filter-repo` (or just rotate and move on — the bot already has it).
- **`git add .` is dangerous.** Always run `git status` first to see what will be staged. Better: `git add <specific-paths>`.
- **Pull before you push** if you work across machines: `git pull --rebase origin main`. Without rebase, merges create clutter.
- **Never force-push to `main`.** Rewrites history that others depend on. Force-pushing your own feature branch is fine if no one else is using it.
- **Render auto-deploys on push to `main`.** Once Render is connected to your GitHub repo, every merge to `main` triggers a deploy. Treat `main` as "production." Break feature branches, not `main`.
- **`.gitkeep` is convention only.** Git doesn't recognize the name. The file is just a placeholder to make the folder non-empty. Any other filename works the same.
- **The `--global` flag for `git config` persists across all repos.** Without it, the setting is local to the current repo. Pick deliberately.

---

## Revise (3-line summary)

1. Always confirm `git rev-parse --show-toplevel` points at your project root — a misplaced `.git/` in the home folder will silently treat your entire computer as a repo.
2. `.gitignore` blocks secrets, venv, build artifacts, OS junk; Git ignores empty folders entirely — for this project we let folders appear naturally when real content lands instead of using `.gitkeep` placeholders.
3. Workflow: branch off `main` per module (`feature/<name>`), commit small logical steps with Conventional Commit messages (`feat:`, `fix:`, `chore:`), merge back via PR when the module is done and tested.
